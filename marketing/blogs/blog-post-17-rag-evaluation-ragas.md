## The Flaw with Manual RAG Evaluation

When building RAG systems, developers typically ask the bot 10-15 questions and read the responses. If the answers look plausible, they ship it. 

The problem with LLMs is that they are designed to be highly persuasive. A hallucinated response will often sound perfectly logical. Furthermore, manual testing cannot isolate whether a failure occurred because:
- The vector database retrieved the wrong chunks (Retrieval Failure).
- The LLM ignored the retrieved chunks and hallucinated (Generation Failure).

To fix this, we must decouple the retrieval evaluation from the generation evaluation.

## The 4 Core RAGAS Metrics

RAGAS leverages an "LLM-as-a-judge" methodology to grade your RAG pipeline across four core mathematical vectors:

| Metric | Focus Area | What it measures |
|---|---|---|
| **Context Precision** | Retrieval | Did the vector search rank the most relevant documents at the very top? |
| **Context Recall** | Retrieval | Did the retrieved chunks contain *all* the information needed to answer the question? |
| **Faithfulness** | Generation | Is the generated answer entirely derived from the retrieved context, or did the LLM hallucinate external facts? |
| **Answer Relevance** | Generation | Does the generated answer directly address the user's prompt without rambling? |

> [!IMPORTANT]  
> A high Context Recall score but a low Faithfulness score indicates that your vector database is working perfectly, but your LLM prompt needs stricter grounding constraints to prevent it from hallucinating outside the context window.

## Step 1: Constructing the Ground Truth Dataset

To evaluate a RAG system, you need a dataset consisting of:
1. `question`: The user prompt.
2. `contexts`: The actual text chunks retrieved by your system.
3. `answer`: The response generated by your LLM.
4. `ground_truth`: The scientifically correct answer written by a human expert.

```python
# sample_eval_dataset.py
dataset = {
    "question": ["What is the penalty for late tax filing under Section 42?"],
    "contexts": [["Section 42 states that late filing incurs a 5% penalty per month..."]],
    "answer": ["The penalty is 5% per month, up to a maximum of 25%."],
    "ground_truth": ["Under Section 42, late filing incurs a 5% monthly penalty."]
}
```

Notice the hallucination? The LLM added "up to a maximum of 25%," which was not in the retrieved context. RAGAS will catch this.

## Step 2: Implementing the RAGAS Pipeline in Python

Using the `ragas` library, we can programmatically score this dataset. 

```python
from datasets import Dataset 
from ragas import evaluate
from ragas.metrics import (
    faithfulness,
    answer_relevancy,
    context_recall,
    context_precision,
)
import os

os.environ["OPENAI_API_KEY"] = "sk-..." # RAGAS uses OpenAI to grade the outputs

# 1. Convert our dictionary into a HuggingFace Dataset
eval_dataset = Dataset.from_dict(dataset)

# 2. Run the evaluation
result = evaluate(
    eval_dataset,
    metrics=[
        context_precision,
        context_recall,
        faithfulness,
        answer_relevancy,
    ],
)

print(result)
# Output: {'context_precision': 1.0, 'context_recall': 1.0, 'faithfulness': 0.5, 'answer_relevancy': 0.98}
```

In this evaluation, `faithfulness` dropped to `0.5` because the LLM generated claims (the 25% cap) that could not be traced back to the `contexts`.

## Step 3: Integrating Evaluation into CI/CD

Evaluation should not be a one-off task. At NimbleSL, whenever a developer tweaks the prompt template or adjusts the chunking strategy in the vector database, a GitHub Action automatically runs the RAGAS evaluation against a golden dataset of 50 edge-case questions.

If the aggregate `faithfulness` score drops below 0.90, the pull request fails automatically.
